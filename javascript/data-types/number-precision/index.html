数字精度与安全整数

结论先行：JavaScript 只有一种数字类型 Number，整数和小数都用 IEEE 754 双精度 64 位浮点数 存储。这一个设计带来两类坑：

  • 小数算不准：0.1 + 0.2 === 0.30000000000000004，不等于 0.3。
  • 大整数存不下：超过 2^53 - 1 的整数无法被精确表示，9007199254740993 会变成 9007199254740992。

理解这两个坑，先得看清这 64 位是怎么分配的。

一个 Number 占 64 位​

IEEE 754 双精度把 64 个二进制位分成三段：

部分位数作用
符号位 (sign)1正还是负
指数位 (exponent)11小数点放在哪，决定数量级
尾数位 (mantissa)52有效数字本身

任何数都被表示成 ± 1.xxxxx × 2^n 的形式。其中 1.xxxxx 这部分的 xxxxx 占 52 位，再算上整数位那个隐含的 1，有效精度一共 53 位二进制。这个 53 是后面一切问题的根源。

信息

生活类比：一把刻度固定的尺子。这把尺子总共只有 53 格刻度。量小东西时刻度细，量大东西时被迫拉长、刻度变粗。所以无论小数还是大整数，一旦需要的精度超过 53 格，尺子就量不准了——要么塞不下小数的无限位，要么分不清两个挨得很近的大整数。

小数为什么算不准​

计算机里小数要先转成二进制。十进制 0.1 转二进制是 0.0001100110011... 无限循环，53 位之后只能舍入截断，于是 0.1 存进去时已经有了微小误差。0.2 同理。两个带误差的数相加，误差累积，结果就成了 0.30000000000000004。

0.1 + 0.2; // 0.30000000000000004
0.1 + 0.2 === 0.3; // false
(0.1 + 0.2).toFixed(2); // "0.30"  仅用于显示，不改变底层值
注意

这不是 JS 的 bug，而是所有用 IEEE 754 的语言（Java、Python、C 等）的共同行为。只要是二进制浮点，就躲不开。

怎么判断「约等于」​

不要直接 === 比两个小数，而是看它们的差是否小于一个极小阈值 Number.EPSILON（2^-52 ≈ 2.22e-16，是 1 与比它大的最近浮点数之间的间隔）：

function numberEqual(a, b) {
  return Math.abs(a - b) < Number.EPSILON;
}
numberEqual(0.1 + 0.2, 0.3); // true

大整数为什么存不下：安全整数​

尾数只有 53 位有效精度，意味着能被精确表示的整数上限是 2^53。一旦超过，相邻整数之间的「刻度」就大于 1，有些整数落在刻度缝里，只能四舍五入到最近的可表示值。

Math.pow(2, 53); // 9007199254740992
Math.pow(2, 53) === Math.pow(2, 53) + 1; // true！9007199254740992 和 ...993 撞成同一个值
9007199254740993; // 9007199254740992  字面量直接就被改了

所以 JS 定义了安全整数的概念：在 [-(2^53 - 1), 2^53 - 1] 范围内的整数，既能被精确表示，也不会和别的整数撞值。

常量值含义
Number.MAX_SAFE_INTEGER9007199254740991 (2^53 - 1)最大安全整数
Number.MIN_SAFE_INTEGER-9007199254740991最小安全整数
提示

为什么是 2^53 - 1 而不是 2^53？因为「安全」要求双向唯一：不光这个整数自己能存下，它 +1 后也得和它不同。2^53 自己存得下，但 2^53 + 1 已经存不下（会变回 2^53），所以 2^53 不算安全，安全上界取到 2^53 - 1。

判断整数与安全整数​

Number.isInteger(42); // true   是不是整数
Number.isInteger(42.0); // true  42.0 在 JS 里就是 42
Number.isInteger(42.5); // false

Number.isSafeInteger(2 ** 53 - 1); // true
Number.isSafeInteger(2 ** 53); // false  超出安全范围

数字的边界值一览​

常量近似值含义
Number.MAX_VALUE1.7976931348623157e308能表示的最大正数，再大就成 Infinity
Number.MIN_VALUE5e-324能表示的最接近 0 的正数（注意不是最小负数）
Number.MAX_SAFE_INTEGER9007199254740991最大安全整数 2^53 - 1
Number.MIN_SAFE_INTEGER-9007199254740991最小安全整数
Number.EPSILON2.220446049250313e-161 与下一个浮点数的间隔，做精度比较用
Infinity / -Infinity—超过 MAX_VALUE 或除以 0 的结果
信息

别被名字误导：Number.MIN_VALUE 不是最小的负数，而是最小的正数（最接近 0）。真正的「最负的数」是 -Number.MAX_VALUE。

解决办法​

小数精度：

  1. Number.EPSILON 容差比较：判断两数是否相等用「差的绝对值 < EPSILON」，别直接 ===。
  2. 放大成整数再算：金额计算把「元」乘 100 当「分」算，整数运算无误差，最后再除回去。
  3. 专用十进制库：金融等高精度场景用 decimal.js、big.js，内部用字符串/十进制运算彻底避开浮点。

大整数：用 BigInt

ES2020 引入 BigInt，专门表示任意精度整数，不受 2^53 限制。字面量在数字后加 n：

9007199254740993n; // 9007199254740993n  不再丢精度
9007199254740991n + 2n; // 9007199254740993n
2n ** 64n; // 18446744073709551616n  随便多大都行
注意

BigInt 与 Number 不能直接混合运算（1n + 1 报 TypeError），需先显式转换。它也只表示整数，不能表示小数。大整数 ID、雪花算法、超大数计算适合用它。
