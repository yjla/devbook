# 四次挥手

1. 第一次挥手

   - 报文详情：客户端向服务端发送一个报文，标志位为 `FIN=1`。
   - 状态变化：客户端发出报文后，进入 `FIN_WAIT_1` 状态。

2. 第二次挥手

   - 报文详情：服务端向客户端返回一个报文，标志位为 `ACK=1`。
   - 状态变化：服务端发出报文后，进入 `CLOSED_WAIT` 状态；客户端收到报文后，进入 `FIN_WAIT_2` 状态。

3. 第三次挥手

   - 报文详情：服务端再次向客户端返回一个报文，标志位为 `FIN=1`。
   - 状态变化：服务端发出报文后，进入 `LAST_ACK` 状态。

4. 第四次挥手

   - 报文详情：客户端再次向服务端发送一个报文，标志位为 `ACK=1`。

   - 状态变化：客户端发出报文后，进入 `TIME_WAIT` 状态。服务端收到报文后，进入 `CLOSE` 状态。客户端在经过 `2MSL` 一段时间后，进入 `CLOSE` 状态。




## 为什么是四次

挥手比握手的次数多了一次，这是因为服务端的 `ACK` 和 `FIN` 是分开发送的。`FIN` 仅表示自身不再发送数据，但还能接收数据。因此，在第三次挥手前，服务端仍能向客户端发送数据。



## 为什么需要等待 2MSL

等待 `2MSL` 是为了保证最后一次挥手，客户端发送的 `ACK` 能够到达服务端。最后一次挥手的 `ACK` 报文可能会丢失。如果没有等待时间，客户端发送 `ACK` 后直接关闭，那么服务端可能会没有收到 `ACK`，迟迟不能关闭，且不停重发 `FIN` 报文。

有了等待时间，客户端等待的是有没有重发的 `FIN` 报文。如果有，则 `2MLS` 重新计时。
