# 三次握手

1. 第一次握手

   - 报文详情：客户端向服务端发送一个报文，标志位为 `SYN=1`；序列号为客户端初始序列号 `client_isn `；该报文不包含应用层数据。

   - 状态变化：一开始，客户端和服务端都处于 `CLOSED` 状态。握手前，服务端主动监听某个端口，处于 `LISTEN` 状态。第一次握手后，客户端处于 `SYN-SENT` 状态。

2. 第二次握手

   - 报文详情：服务器端向客户端返回一个报文，标志位为 `SYN=1`，`ACK=1`；序列号为服务端初始序列号 `server_isn`，确认应答号为 `client_isn + 1`；该报文也不包含应用层数据。

   - 状态变化：第二次握手后，服务端进入 `SYN-RCVD` 阶段

3. 第三次握手

   - 报文详情：客户端向服务端再发送一个报文，标志位为 `ACK=1`；确认应答号为 `server_isn + 1`；这次报文可以携带数据。

   - 状态变化：客户端发出报文后，进入 `ESTABLISHED` 状态；服务端收到报文后，也进入 `ESTABLISHED` 状态。




## 为什么是三次

主要原因是为了防止重复连接，减少不必要的资源开销，能帮助双方同步初始化序列号。

假设有这样一种场景：

1. 客户端发送了第一个请求连接，但由于网络拥堵，服务端并没有收到。

2. 由于服务端没有收到，也就不会返回响应报文。于是，客户端再次发送请求。

3. 在没有第三次握手的情况下，服务端收到了两次请求，因此会建立两次 TCP 连接；


有第三次握手的情况下，客户端会根据第二次握手服务端返回的报文，确认是否为历史连接。如果是，则第三次握手发送的报文的标志位为 `RST=1`，表示中止连接；否则，则成功建立连接。
