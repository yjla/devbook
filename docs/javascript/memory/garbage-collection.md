# 垃圾回收机制

JavaScript 的内存由引擎**自动回收**,开发者不用手动释放。核心问题只有一个:**怎么判断一块内存还要不要**?答案是看「可达性」——从根出发还能访问到的对象就是活的,访问不到的就回收。

## 可达性 (reachability)

GC 从一组**根对象** (全局对象、当前调用栈里的变量等) 出发,沿着引用一路能访问到的对象,都算「可达」、不回收;访问不到的就是垃圾。

```js
let user = { name: 'Tom' }; // 对象可达:被 user 引用
user = null; // 原对象不再可达 → 下次 GC 回收
```

## 两种基础算法

### 引用计数

给每个对象记一个「被引用次数」,降到 0 就回收。简单,但有**致命缺陷:循环引用回收不掉**。

```js
function leak() {
  const a = {};
  const b = {};
  a.ref = b;
  b.ref = a; // a、b 互相引用,计数永远 ≥ 1
}
leak(); // 函数结束后 a、b 已无用,但引用计数法回收不了
```

### 标记清除 (mark-and-sweep)

现代引擎的主流方案。分两步:**标记**——从根遍历,标记所有可达对象;**清除**——回收所有没被标记的。因为判断的是「能不能从根到达」而非「被引用几次」,所以上面的循环引用能被正确回收 (`a`、`b` 都从根不可达)。

## V8 的分代回收

V8 基于一个经验规律——**大部分对象活得很短**——把堆分成两代,用不同策略:

```mermaid
flowchart LR
  N["新生代 New Space<br/>小、对象存活短"] -->|多次 GC 后仍存活<br/>晋升| O["老生代 Old Space<br/>大、对象存活久"]
```

- **新生代**:用 **Scavenge** 算法。空间一分为二 (From / To),只把存活对象复制到另一半,然后整体清空原半区。速度快,代价是浪费一半空间——但新生代本来就小。
- **老生代**:用**标记清除 + 标记整理**。标记整理会在清除后把存活对象往一端挪,消除内存碎片。
- **晋升**:新生代对象经历多次 GC 仍存活,就被移到老生代。

## 减少卡顿的优化

老生代体量大,一次全停顿 (Stop-The-World) 标记会冻结主线程造成卡顿。V8 用这些手段摊薄:

- **增量标记**:把一次大标记拆成多个小步,穿插在 JS 执行之间。
- **并发 / 并行回收**:用额外线程在后台做标记和清理,尽量不阻塞主线程。

## 内存泄漏:GC 也救不了的情况

只要对象**仍然可达**,GC 就不会回收——这正是内存泄漏的根源。常见场景见 [内存管理](./memory-management):意外的全局变量、被遗忘的定时器、不合理的闭包、游离的 DOM 引用。
