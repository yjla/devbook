# JS 引擎如何执行代码

**JS 既不是纯解释执行,也不是像 C 那样提前编译成机器码,而是「先编译成字节码解释执行,再把热点代码即时 (JIT) 编译成机器码」。** 理解这条流水线,要先分清两组容易混的概念:

- **字节码 vs 机器码**:回答「编译成什么」——给虚拟机跑的中间码,还是给 CPU 直接跑的原生码。
- **AOT vs JIT**:回答「什么时候编译」——运行前编译好,还是运行中现编译。

这两组是**不同维度**,常被混成「AOT 字节码」一类的说法,其实可以自由组合。

## 字节码与机器码:区别在「谁来执行」

| | 字节码 (bytecode) | 机器码 (machine code) |
|---|---|---|
| 执行者 | **虚拟机 / 引擎** 翻译后执行 | **CPU** 直接执行 |
| 平台相关 | 否,跨平台(一份到处跑) | 是,绑定 CPU 架构 (x86 / ARM) |
| 速度 | 慢一层(要转译) | 最快(硬件直接跑) |
| 例子 | V8 Ignition 字节码、JVM `.class`、Python `.pyc` | `.exe`、编译好的 `.so` / `.o` |

字节码是**中间产物**:它不针对任何真实 CPU,而是针对一台「虚拟的机器」(V8、JVM、Python VM)。这台虚拟机再把字节码翻译成当前硬件的机器码来跑。

一条加法,直观感受差异:

```text
# 字节码(抽象、跨平台,不关心有几个寄存器)
Ldar a        # 把 a 加载到累加器
Add  b        # 加上 b

# x86-64 机器码对应的汇编(绑定具体寄存器和架构)
mov eax, [rbp-4]
add eax, [rbp-8]
```

字节码只说「把两个数相加」;机器码必须落到 `eax` 这种**具体 CPU 资源**上。**字节码把「平台差异」推迟到虚拟机里处理**,这正是它能跨平台、机器码不能的根本原因。

:::tip 形象记忆
字节码像**菜谱**(「热油、下蒜、翻炒」),抽象、谁都看得懂、换个厨房照样做;机器码像**针对你家这台灶的精确操作**(「左边第二个旋钮转到 3 档」),最直接最快,但换个灶台就不对了。虚拟机就是那个「照着菜谱、用你家灶具操作」的厨师。
:::

## AOT 与 JIT:区别在「什么时候编译」

- **AOT (Ahead-Of-Time,提前编译)**:**运行之前**就编译好。
- **JIT (Just-In-Time,即时编译)**:**运行过程中**,发现反复执行的「热点代码」再把它编译成机器码。

| | AOT | JIT |
|---|---|---|
| 编译时机 | 运行前(构建 / 安装时) | 运行中(检测到热点时) |
| 启动速度 | 快(直接跑) | 慢(有「预热」过程) |
| 优化能力 | 只能静态分析,看不到真实数据 | **能用运行时信息激进优化**(按实际类型特化、内联,失效再「去优化」) |
| 运行时开销 | 无编译器开销,内存小 | 要带编译器,占 CPU / 内存 |
| 典型 | C / C++ / Rust / Go、GraalVM Native Image | V8、JVM 的热点编译 |

JIT 的精髓是**「赌」**:它观察到某个函数总是被传入数字,就赌它「永远是数字」,编译出一份只处理数字、极快的机器码;一旦哪次真传了字符串,赌输了就**去优化 (deoptimization)**,退回字节码重新来。AOT 编译时还没运行,看不到这些,只能保守编译。

## V8 执行 JS 的完整流水线

把上面两组概念套到 V8 上,JS 的执行是这样一条流水线:

```mermaid
flowchart LR
    A["源码"] --> B["解析<br/>Parser"]
    B --> C["AST<br/>抽象语法树"]
    C --> D["Ignition<br/>生成并解释字节码"]
    D -->|"热点代码<br/>(反复执行)"| E["TurboFan<br/>JIT 编译成机器码"]
    E -.->|"类型假设失效<br/>去优化"| D
```

1. **解析 (Parse)**:源码 → AST (抽象语法树),一棵描述代码结构的树。
2. **生成字节码 (Ignition)**:把 AST 编译成**字节码**,然后**解释执行**。这一步快、内存小,保证 JS 启动迅速。
3. **JIT 编译 (TurboFan)**:V8 在解释字节码时顺便统计哪些函数被反复调用 (热点)。对热点函数,基于观察到的运行时类型,**JIT 编译成高度优化的机器码**,后续直接跑机器码。
4. **去优化 (Deopt)**:如果某次调用打破了之前的类型假设 (比如一直传 number 的函数突然传了 string),丢弃这份机器码,退回字节码重新解释,必要时重新优化。

:::info 为什么 JS 不直接 AOT 成机器码
因为 JS 太**动态**:变量类型、对象结构都能运行时改变,编译期根本不知道 `a + b` 是数字加法还是字符串拼接。只能先解释着跑、边跑边观察类型,等摸清规律 (热点 + 稳定类型) 再 JIT。这也是为什么「**保持变量类型稳定、对象结构固定**」能让 V8 优化得更好——别让它的赌注落空。
:::

## 和其他语言对比

同样一段逻辑,不同语言的编译执行路线完全不同:

| 语言 | 路线 | 说明 |
|---|---|---|
| **C / C++ / Rust** | 源码 ──AOT──▶ 机器码 | 无字节码、无虚拟机,启动即跑、最快,但产物绑定平台 |
| **Java** | 源码 ──AOT──▶ 字节码 ──JIT──▶ 机器码 | `javac` 提前编译成字节码,JVM 运行时把热点 JIT 成机器码 |
| **JavaScript** | 源码 ──▶ 字节码 ──JIT──▶ 机器码 | 同上,但字节码是引擎在运行时现生成的,不预先产出文件 |
| **Python (CPython)** | 源码 ──▶ 字节码 ──解释──▶ (无 JIT) | 编译成 `.pyc` 字节码后纯解释执行,所以相对慢 |

:::note
注意 Java 那行:`javac` 把源码编译成字节码,这一步**本身就是 AOT**——只是产物是字节码而非机器码。可见「AOT」和「字节码」确实是两个维度,能任意搭配。
:::

## 一句话总结

> **字节码 vs 机器码**是「跑在虚拟机上还是直接跑在 CPU 上」;**AOT vs JIT** 是「运行前编译还是运行中编译」。JS 走的是「字节码解释 + 热点 JIT」的混合路线:用字节码换**快启动**和**跨平台**,用 JIT 换**运行时极致优化**——代价是动态性太强、无法像 C 那样 AOT 成机器码。
