# 从输入 URL 到页面呈现

1. 解析URL：检查传输协议和请求资源的路径是否合法

2. 缓存判断：检查强缓存，如果命中直接使用，否则向服务器发起请求

3. DNS 解析：由于我们输入的是域名，而数据包是通过 IP 地址传给对方的，因此我们需要使用 DNS 将域名解析为对应的 IP 地址。得注意的是，浏览器提供了 DNS 数据缓存功能。即如果一个域名已经解析过，那会把解析的结果缓存下来，下次处理直接走缓存，不需要经过 DNS 解析。

4. 建立 TCP 连接：进行 TCP 三次握手，如果使用的是 HTTPS，还需要进行 TLS 四次握手

5. 发送 HTTP 请求：浏览器向服务器端发送 HTTP 请求，包含请求行，请求头和请求体。

6. 收到 HTTP 响应：浏览器收到服务器发出的 HTTP 响应，包含响应行、响应头和响应体。服务器根据请求找到这个被请求的文档（如果这个文档不存在，那么将返回一个404响应）， 并把这个文档通过HTTP发送给浏览器。

7. 解析与渲染，关键渲染路径（Critical Rendering Path）五步骤：

   1. 构建文档对象模型（DOM）树：处理 HTML 元素并构造 DOM 树。DOM 树描述了文档的内容，反映了不同元素之间的关系和层次结构。`<html>` 元素是 DOM 树的根节点，嵌套在其他元素中的元素是子节点。DOM 节点的数量越多，构建 DOM 树所需的时间就越长。

   2. 构建 CSS 对象模型（CSSOM）树：处理 CSS，根据继承性和层叠性计算每个节点的具体样式并构建 CSSOM 树。浏览器遍历 CSS 中的每个规则集，根据CSS选择器创建具有父、子和兄弟关系的节点树。

      需要注意：

      - CSS 加载不会阻塞 DOM 解析，因为 CSSOM 树的合成与 DOM 树的合成无关
      - CSS 加载会阻塞页面渲染，因为如果不阻塞，页面首先会渲染出一个样式，CSS 加载后又渲染出另一种样式，这增加了渲染成本
      - CSS 加载会阻塞 JS 的执行，因为 JS 脚本的内容可能是获取元素的样式，如果 CSS 未加载完成，则 JS 无法执行
      - JS 阻塞 DOM 解析，因为 JS 脚本的内容可能是对 DOM 元素进行操作。在解析 HTML 的过程中，如果发现 `<script>` 元素，就暂停解析，把网页渲染的控制权转交给 JS 引擎，等 JS 脚本加载并执行完毕后，再把控制权交还渲染引擎

   3. 合成渲染树（Render Tree）：将 DOM 树和 CSSOM 树组合成一个渲染树，像设置了 `display: none` 样式的结点，就不会出现在渲染树上。

   4. 布局（Layout）：构建渲染树后，开始布局，计算每个节点的几何体。布局是确定呈现树中所有节点的大小和位置，以及确定页面上每个对象的大小和位置的过程。第一次确定节点的大小和位置称为布局。随后对节点大小和位置的重新计算称为回流（reflow）。

   5. 绘制（paint）：将在布局阶段计算的每个框转换为屏幕上的实际像素。

8. 断开连接：判断请求头或响应头中是否包含 `Connection: keep-alive` 字段，如果有，TCP 连接会一直保持；如果 `Connection: close` 则TCP 四次挥手断开连接
