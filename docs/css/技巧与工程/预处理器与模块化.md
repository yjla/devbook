# 预处理器与模块化

原生 CSS 缺少变量、嵌套、逻辑复用，且默认全局作用域容易冲突。预处理器解决前者，模块化方案解决后者。

## CSS 预处理器

**痛点**：原生 CSS 没有变量（颜色改一处要改多处）、没有嵌套（选择器层层重复）、没有函数和复用机制（重复样式只能复制粘贴）、没有真正的模块导入。

Sass/Less 这类预处理器用一套增强语法写样式，再**编译成普通 CSS**，补齐了这些能力：

- **变量**：统一管理颜色、间距、字号
- **嵌套**：选择器按 DOM 层级嵌套书写，更直观
- **mixin**：把一组样式封装成可复用的「样式函数」
- **函数 / 运算**：颜色计算、尺寸运算
- **模块化导入**：`@use` / `@import` 拆分和组合文件

### Sass vs Less

| 维度 | Sass (SCSS) | Less |
|------|-------------|------|
| 变量符号 | `$primary` | `@primary` |
| 语法 | `.scss`（兼容 CSS）/ `.sass`（缩进） | `.less`（兼容 CSS） |
| 逻辑能力 | 更强（条件、循环、函数完善） | 较弱 |
| 运行环境 | 通常编译时（Dart Sass） | 可编译时，也支持浏览器端运行时编译 |
| 生态 | 更主流，社区更大 | 早期 Bootstrap 带火，现在份额下降 |

:::info SCSS 与 Sass
Sass 有两种语法：`.sass` 用缩进、省略大括号和分号；`.scss` 是 CSS 的超集，写法和 CSS 一致只是多了增强功能。现在基本都用 `.scss`。
:::

### Sass 示例（嵌套 + 变量 + mixin）

```scss
// 变量
$primary: #3498db;
$radius: 6px;

// mixin：可带参数的样式片段
@mixin flex-center($direction: row) {
  display: flex;
  flex-direction: $direction;
  align-items: center;
  justify-content: center;
}

.card {
  @include flex-center(column);    // 调用 mixin
  padding: 16px;
  border-radius: $radius;

  // 嵌套
  .title {
    color: $primary;
    font-weight: bold;
  }

  // & 引用父选择器，拼接伪类
  &:hover {
    box-shadow: 0 2px 8px rgba($primary, 0.3);   // 内置颜色函数
  }
}
```

编译后会展开成 `.card`、`.card .title`、`.card:hover` 等普通 CSS 规则。

## 原生 CSS 变量

原生 CSS 也有变量，称为**自定义属性**：用 `--x` 声明，用 `var()` 读取。

```css
:root {
  --primary: #3498db;
  --gap: 16px;
}

.btn {
  color: var(--primary);
  padding: var(--gap);
  background: var(--bg, #fff);   /* 第二个参数是兜底默认值 */
}
```

它和预处理器变量是**两种东西**，区别很关键：

| 对比 | 原生 CSS 变量 (`--x`) | 预处理器变量 (`$x` / `@x`) |
|------|----------------------|---------------------------|
| 生效时机 | **运行时**，存在于浏览器 | **编译时**，编译后即消失 |
| 能否被 JS 读改 | 能，`getComputedStyle` / `style.setProperty` | 不能，运行时已经是写死的值 |
| 作用域 | 跟随 CSS 层叠和继承，可在不同选择器下覆盖 | 词法作用域，按代码块 |
| 响应媒体查询 / 主题切换 | 能，改一个变量全局生效 | 不能，需重新编译 |

```javascript
// JS 动态修改原生 CSS 变量，常用于主题切换
document.documentElement.style.setProperty('--primary', '#e74c3c')
```

:::tip 建议
两者可以共存：用预处理器组织代码（嵌套、mixin），用原生 CSS 变量做需要运行时切换的主题色、暗色模式。
:::

## CSS 模块化方案

原生 CSS 的类名是**全局**的，不同文件里同名的 `.title` 会互相覆盖。模块化方案就是为了解决全局污染和命名冲突。

### CSS Modules

把样式文件当作模块导入，构建工具自动把类名编译成带 hash 的唯一名字，从而实现**局部作用域**。

```css
/* Button.module.css */
.btn {
  color: #3498db;
}
```

```javascript
import styles from './Button.module.css'

function Button() {
  // styles.btn 实际是 "Button_btn__a1b2c" 这样的 hash 类名
  return <button className={styles.btn}>OK</button>
}
```

不同组件即使都写 `.btn`，编译后类名也不同，天然不冲突。

### BEM 命名规范

不依赖工具，靠**命名约定**避免冲突。结构是 `Block__Element--Modifier`：

- **Block**：独立的功能块，如 `card`
- **Element**：块的组成部分，用 `__` 连接，如 `card__title`
- **Modifier**：状态或变体，用 `--` 连接，如 `card--featured`

```css
.card { }
.card__title { }
.card__button { }
.card--featured { }              /* 变体 */
.card__button--disabled { }      /* 元素的状态 */
```

```html
<div class="card card--featured">
  <h3 class="card__title">标题</h3>
  <button class="card__button card__button--disabled">按钮</button>
</div>
```

通过命名把作用域写进类名里，全局唯一，但完全靠人维护。

### 其他思路

- **CSS-in-JS**（styled-components、Emotion）：在 JS 里写样式，作用域和动态能力都靠 JS 运行时，能根据 props 动态生成样式，代价是运行时开销。
- **Tailwind**（原子化 CSS）：不写自定义类，直接用 `flex`、`p-4`、`text-blue-500` 这类预设原子类组合，没有命名问题，但 HTML 类名会很长。

### 方案对比

| 方案 | 全局污染 | 作用域隔离 | 动态能力 | 心智成本 |
|------|----------|-----------|----------|----------|
| 原生 CSS | 有 | 无 | 弱（仅 CSS 变量） | 低 |
| BEM | 靠约定规避 | 命名隔离 | 弱 | 中（需严格遵守命名） |
| CSS Modules | 无 | 编译期 hash | 中（配合 CSS 变量） | 低 |
| CSS-in-JS | 无 | 运行时 | 强（基于 props） | 中高 |
| Tailwind | 无 | 原子类无命名 | 中 | 中（需记忆类名） |

:::tip 建议
React 工程里 CSS Modules 是省心的默认选择：零运行时、作用域隔离、写法接近原生 CSS。需要强动态样式时再考虑 CSS-in-JS，追求开发速度和一致性时考虑 Tailwind。
:::
