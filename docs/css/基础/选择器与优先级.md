# 选择器与优先级

当多条 CSS 规则命中同一个元素的同一个属性时，谁生效由**优先级 (specificity)** 决定；优先级相同则看书写顺序，后写的覆盖先写的。

## 常见选择器类型

| 选择器 | 写法 | 含义 |
|--------|------|------|
| 元素选择器 | `div` | 匹配所有 `div` 标签 |
| 类选择器 | `.box` | 匹配 `class="box"` 的元素 |
| ID 选择器 | `#app` | 匹配 `id="app"` 的元素 |
| 属性选择器 | `[type="text"]` | 匹配带指定属性的元素 |
| 后代选择器 | `.list li` | `.list` 内任意层级的 `li` |
| 子代选择器 | `.list > li` | `.list` 的直接子级 `li` |
| 相邻兄弟选择器 | `h2 + p` | 紧跟 `h2` 之后的第一个 `p` |
| 通用兄弟选择器 | `h2 ~ p` | `h2` 之后的所有同级 `p` |
| 通用选择器 | `*` | 匹配所有元素 |
| 伪类 | `a:hover` | 匹配元素的特定状态 |
| 伪元素 | `p::before` | 匹配元素生成的虚拟内容 |

```css
.list > li        /* 直接子级 */
.list li          /* 任意后代 */
h2 + p            /* 相邻的下一个兄弟 */
input[type="checkbox"]:checked   /* 属性 + 伪类组合 */
```

## 优先级计算规则

优先级用一个四元组 `(a, b, c, d)` 表示，从高到低逐位比较：

| 类别 | 对应位 | 权重 | 包含 |
|------|--------|------|------|
| 内联样式 | a | 1000 | `style="..."` |
| ID 选择器 | b | 100 | `#app` |
| 类 / 属性 / 伪类 | c | 10 | `.box`、`[type]`、`:hover` |
| 元素 / 伪元素 | d | 1 | `div`、`::before` |

规则要点：

- **逐位比较，高位优先**，不会进位。即 `0,1,0,0` (一个 ID) 永远大于 `0,0,99,0` (99 个类)，哪怕后者数值看起来更大。
- `!important` 凌驾于一切普通声明之上，优先级最高。
- 通用选择器 `*` 和**继承来的样式**优先级为 `0`，可被任何具体规则覆盖。
- 优先级完全相同时，**后写的覆盖先写的** (书写顺序)。

:::info
"逐位不进位" 是关键。写成数字 1000/100/10/1 只是方便记忆和近似比较，真正的比较方式是按 `(a, b, c, d)` 四组数从左往右逐组比，而不是把它们加成一个总分。例如 11 个类是 `0,0,11,0`，仍小于 1 个 ID 的 `0,1,0,0`。
:::

## 对比表：算几个选择器的权重

| 选择器 | 内联 | ID | 类/属性/伪类 | 元素/伪元素 | 权重 |
|--------|------|----|----|------|------|
| `*` | 0 | 0 | 0 | 0 | 0 |
| `li` | 0 | 0 | 0 | 1 | 1 |
| `ul li` | 0 | 0 | 0 | 2 | 2 |
| `.list li` | 0 | 0 | 1 | 1 | 11 |
| `a:hover` | 0 | 0 | 1 | 1 | 11 |
| `.list > li.active` | 0 | 0 | 2 | 1 | 21 |
| `#nav .item a` | 0 | 1 | 1 | 1 | 111 |
| `#nav` | 0 | 1 | 0 | 0 | 100 |
| `style="color:red"` | 1 | 0 | 0 | 0 | 1000 |

```html
<a id="nav" class="link" href="#">点我</a>
```

```css
#nav        { color: red; }    /* 100，胜出 */
.link:hover { color: blue; }   /* 11 */
a           { color: green; }  /* 1 */
```

最终文字是红色，因为 ID 选择器权重最高。

## `!important` 的滥用问题

`!important` 会跳过正常的优先级比较直接生效，这让它很容易被当成 "万能补丁"，但代价很大：

- 它破坏了 CSS 自上而下、由弱到强的层叠逻辑，后续想覆盖只能用**另一个 `!important`** 去压，形成军备竞赛。
- 调试时难以追踪某个样式到底来自哪条规则。

:::warning
不要用 `!important` 解决优先级冲突。优先做法是提高选择器自身的特异性 (比如加一层父级类名)，或调整规则的书写顺序。`!important` 只在极少数场景合理，例如覆盖第三方库内联样式、工具类 (utility class) 框架里需要强制生效的原子类。
:::

## 选择器性能：浏览器从右往左匹配

浏览器匹配选择器的方向是**从右往左**，而不是从左往右。

以 `.list li a` 为例，浏览器并不是先找到 `.list` 再往下找，而是：

1. 先找到页面上**所有的 `a`** (最右侧的 "关键选择器")；
2. 对每个 `a` 向上回溯，检查祖先里有没有 `li`，再检查有没有 `.list`；
3. 不匹配就丢弃。

```mermaid
flowchart RL
  A["所有 a 元素 (起点)"] --> B["祖先中有 li?"]
  B --> C["祖先中有 .list?"]
  C --> D["命中，应用样式"]
```

由此得出的实用结论：

:::tip
- **最右侧的选择器应尽量具体** (用类，少用 `*` 或纯标签)，缩小起始候选集。`.list *` 这种以通用选择器结尾的写法代价很高。
- 避免过深的后代嵌套，回溯链越长越慢。
- 现代浏览器选择器性能已经很好，绝大多数页面不必为此过度优化；它更多是写法上的良好习惯，而非性能瓶颈的常见来源。
:::
