# 规范驱动开发

**在让 AI 动手之前，先把「做什么」和「怎么做」写清楚。** 这是规范驱动开发 (Spec-Driven Development，SDD) 的核心思路，也是目前 AI Coding 的主流方法论。

## 早期探索

AI Coding 早期经历了两种截然不同的探索。

**Prompt Engineering** 把注意力放在措辞技巧上——假设问题在于「怎么说」，而不是「说了什么」。随着模型能力提升，精心设计措辞的边际收益越来越小。

**Vibe Coding** 走向另一端：完全放弃结构，顺从 AI 节奏。

> "完全顺从节奏，拥抱指数增长，忘掉代码的存在。" — Andrej Karpathy

快速原型没问题，但「不理解自己代码库」在团队协作和长期维护中是严重的技术债。

两种方式走向了不同的极端，但根本问题是一样的：都没有认真对待给 AI 的信息质量。

## 上下文工程

2025 年，Tobi Lütke (Shopify CEO) 和 Andrej Karpathy 先后表态，倡导用 Context Engineering 替代 Prompt Engineering：

> "Context engineering 比 prompt engineering 更准确，描述的是为任务提供完整上下文的能力。" — Tobi Lütke

核心认知：AI Agent 失败的主要原因不是模型能力不够，而是 **上下文质量不够**。开发者的核心工作从「想办法让 AI 理解我」变成「提前把信息整理清楚」。

## SDD 的诞生

信息不足时，AI 不会停下来等待澄清，而是 **自动填补空白**——填补的方式未必是你想要的。因此，需要将规范 (specification) 作为上下文提前给到 AI，避免它自行填补空白、偏离预期。

2025 年 9 月，GitHub 官方发布 **Speckit**，将这套思路落地为具体工具：在交给 AI 执行之前，先把需求、技术方案和任务边界写清楚。Speckit 被业界广泛采用后，这套实践被命名为 **规范驱动开发 (SDD)**。

:::info Spec 与需求文档的区别
传统需求文档（PRD、设计文档）是写给人看的，开发开始后就归档，很少再被引用。

SDD 的 spec 是 **AI 的上下文输入**，在整个开发过程中持续被引用。因此它的写法也不同：不只描述业务意图，还要明确技术约束、边界条件和「不做的事」——因为 AI 会对没写的部分进行推断。
:::
