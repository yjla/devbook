# 强制 TDD

SDD 回答了「给 AI 什么上下文」，但还有一个问题悬而未决：**如何验证 AI 的实现是正确的？**

TDD 是目前最有效的答案——不是因为测试覆盖率指标好看，而是因为测试是比 `spec.md` 更精确的表达方式：规范描述意图，测试验证行为。

## 为什么需要 TDD

AI 生成代码很快，但它没有内在的验证机制。没有测试，AI 只能根据规范的文字描述来猜测「完成」的样子，而文字描述天然存在歧义。

测试把「完成」定义为可执行的标准：测试通过 = 实现正确。这个反馈闭环对 AI 尤为重要——AI 可以根据测试失败的信息修正实现，而不是凭猜测。

## 红绿循环

TDD 的基本节奏：

1. **红**：先写测试，此时测试必然失败（被测代码还不存在）
2. **绿**：写最少的实现代码让测试通过
3. **重构**：在测试保护下优化代码

AI 执行这个流程时，测试充当「可执行的验收标准」——AI 不需要猜测实现是否正确，测试结果会直接告诉它。

## Superpowers

[Superpowers](https://github.com/obra/superpowers) 由 Jesse Vincent 开发 (2025 年 10 月)，在 SDD 基础上增加了一条硬性规则：**测试必须在实现代码之前写完**。如果 agent 跳过测试直接写了实现，Superpowers 会要求它删除并重来。

这个约束基于提示词层面的行为规则——每次会话启动时规则被注入上下文，agent 在整个会话中遵循它们。除强制 TDD 外，Superpowers 还在此基础上扩展了 Socratic 需求细化、子 Agent 并行执行、结构化 Debug 等能力。

:::tip
测试用例本质上是可执行的规范：`spec.md` 描述需求，测试验证行为，两者共同构成完整的验证闭环。
:::
