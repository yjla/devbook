# AI 运维监控

传统运维依赖人工盯指标、看告警、翻日志。AI 的介入让这个过程从**被动响应**变为**主动感知**：在问题影响用户之前发现它，在告警淹没你之前定位根因。

## 核心能力

### 智能告警分析

原始告警的问题：数量多、噪音高、缺乏上下文。AI 在这里做三件事：

1. **告警聚合**：把同一根因触发的多条告警合并为一个事件
2. **优先级评估**：结合历史数据判断这条告警是否真的需要人工介入
3. **上下文补充**：自动关联相关日志、指标、最近的部署记录

```
传统：20 条告警同时响，不知道从哪开始看
AI：「这 20 条来自同一个根因：Redis 连接池耗尽，
     触发时间与 14:32 的部署高度相关」
```

### 根因分析

AI 能横跨 Metrics、Logs、Traces 三类数据，找到人工难以发现的关联：

```
现象：API 响应时间 p99 从 200ms 升至 2s
AI 分析链路：
  → 数据库查询变慢（traces 显示）
    → 某个查询缺少索引（slow query log）
      → 上周的 migration 删除了该索引（git log 关联）
```

### 自然语言运维

用对话代替命令行和仪表盘操作：

```
你：过去 1 小时 payment-service 有什么异常？
AI：检测到 3 次超时错误（14:15、14:23、14:31），
    均发生在调用第三方支付 API 时，
    建议检查第三方服务状态页

你：帮我生成一个查询过去 24 小时 5xx 错误的 SQL
AI：SELECT ...（直接给出可执行的查询）
```

## 落地路径

### 第一阶段：冷启动

先建知识库，让 AI 了解你的系统：

```markdown
- 整理常见告警及其处理方式（FAQ）
- 记录历史事故的根因和解决方案
- 标注各服务的正常指标范围（基线）
```

没有这些，AI 只能给通用建议，无法针对你的系统做判断。

### 第二阶段：日常接入

AI 参与日常值班流程：

```
告警触发
  → AI 自动分析：是否需要人工介入？根因是什么？
    → 不需要介入：AI 自动记录，聚合到日报
    → 需要介入：AI 生成事故摘要，推送给 On-call
      → On-call 处理时，AI 实时提供诊断建议
```

### 第三阶段：持续改进

每次事故后，把处理过程反馈给 AI，让它学会识别类似模式：

```
事故复盘 → 更新 Runbook → AI 下次遇到同类问题能直接给出解决方案
```

## 与 SLO 结合

AI 运维的目标不是「零告警」，而是保障 SLO（服务等级目标）：

```
SLO：API 可用性 99.9%，p99 延迟 < 500ms

AI 监控角色：
- 预测：当前趋势是否会在未来 1 小时内违反 SLO？
- 预警：提前通知，而不是等到真的违反才告警
- 优先级：影响 SLO 的问题 > 不影响 SLO 的噪音
```

## 注意事项

- **AI 建议需要人工确认**：自动执行运维操作（重启服务、扩容）需要严格的权限控制和审批流程
- **数据质量是前提**：日志不规范、指标缺失，AI 分析效果大打折扣
- **避免过度依赖**：AI 不了解业务上下文，「技术指标正常但用户体验差」的问题仍需人工判断
