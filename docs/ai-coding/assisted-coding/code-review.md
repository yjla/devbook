# AI 辅助代码审查

代码审查是质量保障的核心环节，也是最耗时的环节之一。审查 200-400 行代码通常需要 60-90 分钟，且质量高度依赖审查者的状态和经验。AI 在这里能做的不是替代人工判断，而是**消除机械性工作，让人专注于更高层的决策**。

## 传统审查的问题

- **时间成本高**：认知密集，难以长时间保持专注
- **不一致性**：同一段代码，不同人、不同时间审查结果不同
- **知识孤岛**：审查者不熟悉某个模块时，容易遗漏问题
- **复杂错误被忽视**：容易抓到命名问题，但跨文件的逻辑 bug 难发现

## AI 能做什么

### 自动化的部分

```
代码标准合规性检查     ✅ AI 擅长（规则明确）
安全漏洞扫描          ✅ AI 擅长（模式匹配）
重复/冗余代码识别      ✅ AI 擅长
命名和注释质量         ✅ AI 擅长
依赖影响分析          ✅ AI 可以辅助

业务逻辑正确性         ⚠️  需要人工判断
架构合理性            ⚠️  需要人工判断
产品需求匹配度         ❌ AI 不了解业务背景
```

### 上下文感知审查

好的 AI 审查不只看当前 diff，还需要：
- **历史数据**：这段代码改过几次？上次改动引入了什么问题？
- **系统状态**：这个改动在哪些环境下会被触发？
- **预测性防护**：这个改法在类似场景下历史上出现过什么问题？

## Prompt 工程技巧

直接扔 diff 给 AI 效果一般，结构化的方式更好：

### 分而治之

把大 PR 拆成逻辑块，分别审查：

```markdown
请审查以下代码变更，重点关注：
1. 安全漏洞（SQL 注入、XSS、权限校验）
2. 异常处理是否完整
3. 边界条件

代码：
[具体代码块]
```

### 二次过滤减少误报

先用 LLM 生成问题列表，再用第二次调用过滤假阳性，可减少约 80% 的误报，同时降低约 77% 的 token 消耗。

### 指定角色和标准

```markdown
你是一名专注于 Node.js 后端的高级工程师，
以下是我们团队的代码规范：[规范]
请按 Critical / Warning / Suggestion 三级给出反馈
```

## 主流工具对比

| 工具 | 核心能力 | 适用场景 |
|------|---------|---------|
| GitHub Copilot | PR 摘要、漏洞建议 | 日常 PR 审查 |
| Amazon Q | SAST、密钥检测、IaC 分析 | AWS 生态，安全审查 |
| Snyk Code | 高精度安全漏洞检测 | 安全优先的项目 |
| CodeRabbit | 逐行上下文感知 | 需要细粒度反馈 |
| SonarQube | 规则 + ML 混合 | 企业 CI 门禁集成 |

## 工作流建议

### 分层审查

```
第一层（AI 自动）: 风格、安全、重复代码 → 提交时自动运行
第二层（AI 辅助）: 逻辑分析、影响范围 → PR 创建时生成报告
第三层（人工）: 业务逻辑、架构决策 → 人工 review 只看高层问题
```

### 与 CI 集成

```yaml
# GitHub Actions 示例
- name: AI Code Review
  uses: coderabbitai/ai-pr-reviewer@latest
  with:
    openai_api_key: ${{ secrets.OPENAI_API_KEY }}
    review_comment_lgtm: false  # 不好的地方才留评论
```

AI 代码审查是流程变革，不只是工具替换——需要团队明确哪些决策信任 AI、哪些必须人工把关。
