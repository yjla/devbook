defer 与 async

这俩是 <script> 的两个属性，都用来让脚本异步下载、不阻塞 HTML 解析。区别在「下载完什么时候执行」。

先看普通 script 的问题​

不加任何属性的 <script src>，浏览器解析 HTML 时一旦遇到它，就会停下解析，先下载脚本、执行完，再继续：

解析 HTML ──▶ 遇到 script ──[暂停｜下载｜执行]──▶ 继续解析剩余 HTML

下载和执行这段时间，页面解析完全卡住，后面的内容渲染不出来。脚本越大、网络越慢，白屏越久。传统解法是把 <script> 塞到 <body> 末尾，让它在内容解析完之后才出现。

async：下完就执行，不保证顺序​

<script async src="analytics.js"></script>

async 让脚本下载和 HTML 解析并行，互不阻塞。但脚本一旦下载完成就立即执行，执行的那一刻会打断 HTML 解析。

解析 HTML ──────────────────────▶（解析不停）
        └─下载─┘
              └执行（此刻暂停解析）┘

关键特点：谁先下载完谁先执行，和书写顺序无关。多个 async 脚本之间没有顺序保证，所以它只适合互相独立、不依赖 DOM、也不依赖其他脚本的场景——典型就是统计、广告这类「自己玩自己的」第三方脚本。

defer：解析完再按序执行​

<script defer src="app.js"></script>

defer 同样让脚本下载和 HTML 解析并行，但执行时机推迟到整个 HTML 解析完成之后（DOMContentLoaded 事件之前）。

解析 HTML ───────────────────▶ 解析完毕 ──▶ 执行脚本
        └─下载（并行，不打断解析）─┘

两个关键保证：

  1. 执行时 DOM 已完整——脚本里能放心操作任何元素，不会出现「元素还没解析到、拿到 null」
  2. 多个 defer 按书写顺序执行——有依赖关系（比如库要先于业务代码）也没问题

对比​

下载执行时机多脚本顺序适用
普通阻塞解析下载完立即（阻塞解析）按书写顺序必须立刻执行的内联逻辑
async并行，不阻塞下载完立即（打断解析）不保证独立第三方脚本（统计、广告）
defer并行，不阻塞HTML 解析完之后按书写顺序依赖 DOM、有先后依赖的业务脚本
提示

现代项目里，把脚本放 <head> 加 defer，比放 <body> 末尾更好：脚本能尽早开始并行下载，又保证 DOM 就绪后才执行，首屏更快。而 <script type="module"> 默认就是 defer 行为，无需手动加。

信息

async 和 defer 只对有 src 的外部脚本有效。写在标签里的内联脚本（没有 src）加这两个属性会被忽略，依然是同步立即执行。
